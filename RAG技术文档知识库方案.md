# RAG 技术文档知识库 —— 混搭方案

## 一、方案概览

```
                      ┌──────────────────────────────────┐
                      │          自部署（高频/固定成本）      │
                      │  Embedding + Reranker + 向量库     │
                      │                                  │
  用户提问 ──→  API 层 ──┤                                  ├──→ 最终回答
                      │                                  │
                      │        API调用（按量/灵活切换）      │
                      │           LLM 答案生成             │
                      └──────────────────────────────────┘
```

核心原则：**高频固定成本走自部署，低频按需走API**。

---

## 二、模型选型明细

### 2.1 Embedding 模型：自部署 BGE-M3

| 维度 | 参数 |
|---|---|
| **选型** | BAAI/BGE-M3 |
| **部署方式** | Docker 自建推理服务（vLLM / TEI） |
| **向量维度** | 1024 |
| **选型理由** | 支持稠密+稀疏双路召回，中英双语表现好，模型轻量（约2GB显存），单卡即可跑 |
| **替代方案** | text-embedding-3-small（API，初期快速验证用） |

### 2.2 重排序模型：自部署 BGE-Reranker-v2-m3

| 维度 | 参数 |
|---|---|
| **选型** | BAAI/BGE-Reranker-v2-m3 |
| **部署方式** | 与Embedding同机部署，FastAPI服务 |
| **输入/输出** | 输入 query + Top-20文档 → 输出 Top-5 重排序结果 |
| **选型理由** | 轻量（约1GB显存），技术文档场景下术语精准匹配提升明显，零调用成本 |

### 2.3 向量数据库：自部署 Milvus Lite / Qdrant

| 维度 | 参数 |
|---|---|
| **选型** | Qdrant（起步）/ Milvus（规模扩展） |
| **部署方式** | Docker 自建 |
| **选型理由** | Qdrant 单机性能好、API简洁、支持过滤和混合搜索；Milvus 在千万级以上向量时更优 |
| **起步建议** | Qdrant，Rust实现，内存占用低，文档清晰 |

### 2.4 LLM 答案生成：API 调用

| 模型 | 优先级 | 场景 |
|---|---|---|
| **DeepSeek-V3** | 首选 | 中文技术文档场景，便宜，代码能力顶级 |
| **GPT-4o-mini** | 次选 | 英文文档或需要复杂推理时 |
| **Claude 3.5 Haiku** | 备选 | 长代码生成或需要严格遵循指令时 |

调用策略：统一封装适配层，支持按场景/成本动态路由到不同LLM。

### 硬件需求

| 组件 | 显存需求 | 推荐配置 |
|---|---|---|
| BGE-M3 | ~2GB | 单卡 Tesla T4 即可 |
| BGE-Reranker-v2-m3 | ~1GB | 与Embedding共用 |
| Qdrant | 内存依赖数据量 | 10万文档约需2GB内存 |
| **合计** | **~4-5GB 显存 + 8GB 内存** | T4 / RTX 3060 12G 绰绰有余 |

## 四、数据处理管道

```
爬取HTML ──→ 清洗为Markdown ──→ 智能切片 ──→ 向量化 ──→ 入库
                                       │
                                  保留元数据：
                                  - 来源URL
                                  - 文档标题层级
                                  - 代码块语言标记
                                  - 版本/更新时间
```

### 切片策略

| 文档类型 | 方法 | 参数 |
|---|---|---|
| API参考 | 按"方法/类/属性"边界切 | chunk_size=800, overlap=100 |
| 教程/指南 | 按Markdown标题层级切 | chunk_size=1200, overlap=200 |
| 代码示例 | 示例+说明整体保留 | 以代码块为最小单元 |
| FAQ | 按Q&A对切 | 每个Q&A独立成块 |

### 检索配置

| 参数 | 值 | 说明 |
|---|---|---|
| 粗召回 Top-K | 20 | 向量检索召回 |
| BM25补召回 | 10 | 关键词稀疏检索补充 |
| 重排序后 Top-K | 5 | Reranker精排后送入LLM |
| 混合检索权重 | 向量0.7 / BM25 0.3 | Qdrant原生支持 |

## 五、实施路径

| 阶段 | 内容 | 预计周期 |
|---|---|---|
| **第1周** | 选定文档源（建议RFC Editor），搭建爬虫，爬取全站 | 2-3天 |
| **第2周** | Docker部署 Qdrant + BGE-M3 + Reranker，跑通向量化+检索链路 | 3-4天 |
| **第3周** | 实现RAG问答服务（检索→重排→LLM生成），接DeepSeek-V3 | 3-4天 |
| **第4周** | 构建测试问题集（100题），评估检索召回率与答案质量，迭代切片策略 | 持续 |
| **第5周+** | 扩展第二个文档源，完善监控与增量更新机制 | 按需 |
