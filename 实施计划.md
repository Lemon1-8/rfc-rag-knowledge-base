## 三、关键技术设计

### 3.1 智能切片器（chunker.py）—— Small-to-Big + 按内容类型分治

核心思路：**两层切片 + 四种内容类型分别处理**。

- **小 chunk**（检索层）：300-500 字符，送入 Embedding 做精确语义匹配。越小越精准，不会被无关内容稀释。
- **大 chunk**（上下文层）：整个 h2 section（可能 3000-5000 字符），小 chunk 命中后，将其所属的完整 section 喂给 LLM。上下文完整，不会因为截断丢失关键信息。

流程：`query → 小 chunk Embedding 检索 Top-5 → 映射回各自的大 chunk → 去重 → 大 chunk 送入 LLM`

#### 第一步：解析文档结构 + 内容类型标注

RFC HTML 中遍历所有 heading 标签（h1~h6），构建章节树。同时用 BeautifulSoup 遍历每个章节内的所有元素，为每个文本片段打上类型标签：

```python
ContentType = Enum("ContentType", ["TERM_DEF", "CODE_BLOCK", "LIST", "NARRATIVE"])

# 检测规则
def classify(element) -> ContentType:
    text = element.get_text().strip()

    # 术语定义：以典型定义句开头
    if re.match(r'^(A|An|The)\s+\w+.*\b(is|are|denotes|refers to|means)\b', text):
        return TERM_DEF

    # ABNF 语法 / 代码
    if element.name in ('pre', 'code') or element.find('pre'):
        return CODE_BLOCK

    # 列表结构
    if element.name in ('ul', 'ol') or element.find('ul') or element.find('ol'):
        return LIST

    return NARRATIVE
```

#### 第二步：按章节 + 内容类型做首次分割

每个 h2 section 内部，根据内容类型边界切分为不同的**原子片段**（Atom）。每种类型的处理策略不同：

| 内容类型 | 检测方式 | 首次分割策略 |
|---|---|---|
| **术语定义** | 以 "A/An/The ... is/denotes/refers to/means ..." 开头 | 独立成 Atom，完整保留，不切 |
| **ABNF 语法块** | `<pre>` / `<code>` 标签 | 完整保留，附加章节路径作为上下文说明 |
| **列表** | 连续的 `<ul>` / `<ol>` | 保持列表结构完整，不从中截断 |
| **普通叙述** | 上述都不匹配 | 若超过 500 字符，用 RecursiveCharacterTextSplitter 降级切 |

这样，RFC 7540 第 5 节的切分结果类似于：

```
Section: RFC 7540 / 5. Streams and Multiplexing    ← 大 chunk
  ├── Atom 1 [TERM_DEF]    "A 'stream' is an independent, bidirectional..."
  ├── Atom 2 [LIST]        "Streams have several important characteristics:..."
  ├── Atom 3 [NARRATIVE]   "HTTP/2 endpoints try to resolve the stream..."
  ├── Atom 4 [NARRATIVE]   "There is no requirement that the stream identifier..."
  ├── Atom 5 [NARRATIVE]   "Streams can be terminated by either endpoint..."
  ├── Atom 6 [CODE_BLOCK]  "                            +--------+..."
  └── Atom 7 [NARRATIVE]   "The sender MUST NOT send flow control frames..."
```

#### 第三步：构建两层结构

```python
# 大 chunk —— 整个 h2 section 完整内容
@dataclass
class BigChunk:
    chunk_id: str                    # "rfc7540_sec5"
    document_id: str                 # "rfc7540"
    title: str                       # "RFC 7540 - HTTP/2"
    section_path: str                # "5. Streams and Multiplexing"
    content: str                     # 该 section 的完整 Markdown 原文
    url: str                         # 链接
    small_chunk_ids: list[str]       # 关联的小 chunk ID 列表

# 小 chunk —— 检索用，由 Atom 组装而成
@dataclass
class SmallChunk:
    chunk_id: str                    # "rfc7540_sec5_atom3"
    parent_chunk_id: str             # "rfc7540_sec5"  ← 映射回大 chunk
    content: str                     # 带上下文的精炼文本
    content_type: ContentType        # 内容类型
    section_path: str
    url: str
    content_hash: str                # MD5，幂等控制
```

#### 第四步：从 Atom 组装小 chunk（检索层）

组装策略——邻近的 Atom 合并，但不超过 500 字符上限，且在内容类型边界处尽量不跨类型合并：

```python
def build_small_chunks(atoms: list[Atom], max_size=500) -> list[SmallChunk]:
    chunks = []
    buffer = []
    buffer_len = 0

    for atom in atoms:
        # 跨内容类型时，先 flush 当前 buffer
        if buffer and atom.type != buffer[-1].type:
            chunks.append(flush(buffer))
            buffer, buffer_len = [], 0

        if buffer_len + len(atom.text) <= max_size:
            buffer.append(atom)
            buffer_len += len(atom.text)
        else:
            if buffer:
                chunks.append(flush(buffer))
            # 单个 Atom 超过上限（如长叙述段），用 RecursiveTextSplitter 切
            if len(atom.text) > max_size:
                sub_texts = text_splitter.split_text(atom.text)
                for i, sub in enumerate(sub_texts):
                    chunks.append(SmallChunk(
                        content=f"[{section_path}] {sub}",
                        content_type=atom.type,
                        ...
                    ))
            else:
                buffer, buffer_len = [atom], len(atom.text)

    if buffer:
        chunks.append(flush(buffer))

    return chunks
```

最终每个小 chunk 的内容带上章节锚点：

```
[RFC 7540 / 5.1 Stream States]
A 'stream' is an independent, bidirectional sequence of frames exchanged
between the client and server within an HTTP/2 connection.
```

#### 第五步：分别入库

两个 Qdrant collection：

| Collection | 存储内容 | 用途 |
|---|---|---|
| `rfc_chunks_small` | 小 chunk + 向量 + parent_chunk_id | Dense + BM25 检索，返回 parent_chunk_id |
| `rfc_chunks_big` | 大 chunk（完整 section 原文） | 按 parent_chunk_id 查询，取完整上下文喂 LLM |

检索时只需多一步映射：`小 chunk 检索 → 拿到 parent_chunk_id → 去重 → 查 big collection → 喂 LLM`。查询 big collection 是主键查询，毫秒级。

#### 切片策略总结

| 步骤 | 做什么 | 为什么 |
|---|---|---|
| 1. 解析 + 内容类型标注 | 按 h1~h6 构建章节树，同时标注 TERM_DEF/CODE_BLOCK/LIST/NARRATIVE | 不同内容类型的语义完整性边界不同 |
| 2. 首次分割为 Atom | 按章节 + 内容类型边界分割，代码块和术语定义不切割 | 保护关键语义单元的完整性 |
| 3. 构建两层结构 | BigChunk（完整 section）+ SmallChunk（精炼检索单元） | 检索用小的（精准），生成用大的（完整） |
| 4. 组装小 chunk | 同类型邻近 Atom 合并，上限 500 字符，超限降级切分 | 控制检索粒度，保持类型一致性 |
| 5. 分别入库 | 两个 Qdrant collection，小 chunk 存向量，大 chunk 存原文 | 检索→映射→取原文，只多 1 次主键查询 |

### 3.2 其他核心模块

#### embedding.py
- TEI HTTP 客户端，调用 localhost:8888 的 `/embed` 端点
- `encode(texts: list[str]) -> list[list[float]]`，返回 1024 维向量
- 批量请求（TEI max_client_batch_size=32），httpx AsyncClient

#### reranker.py
- TEI HTTP 客户端，调用 localhost:8889 的 `/rerank` 端点
- `rerank(query, documents, top_k=5)` 返回按分数降序的 top_k

#### llm.py
- 使用 LangChain 的 `ChatOpenAI`，直接兼容 DeepSeek-V3 的 OpenAI 接口
- `generate_stream(prompt) -> AsyncGenerator[str, None]`，逐 chunk yield token
- 配置：`base_url="https://api.deepseek.com/v1"`, `model="deepseek-chat"`

#### vector_store.py
- qdrant-client 库，启动时自动创建两个 collection：
  - `rfc_chunks_small`：小 chunk，含 1024 维向量 + 全文索引 + parent_chunk_id，用于检索
  - `rfc_chunks_big`：大 chunk（完整 section 原文），无向量，仅主键查询，用于取 LLM 上下文
- 利用 Qdrant 1.10+ 原生 TextField 全文索引实现 BM25
- Payload 中包含 content_hash 做幂等控制

#### hybrid_search.py
- 使用 LangChain 的 `QdrantVectorStore` 封装向量检索，BM25 走 Qdrant 原生全文索引
- Dense Top-20 + BM25 Top-10 → RRF 融合（k=60）→ Reranker Top-5
- 在 `rfc_chunks_small` collection 上检索，payload 含 `parent_chunk_id`
- 去重后到 `rfc_chunks_big` 主键查询完整 section 原文

#### rag_chain.py
- 使用 LangChain LCEL (`RunnableSequence`) 链式编排：
  `query → Embedding → HybridSearch → map parent_chunk_id → fetch big chunks → Reranker → PromptTemplate → ChatOpenAI.stream`
- Prompt 模板使用 LangChain `ChatPromptTemplate`，引用格式标注为 `[1]` `[2]`
- SSE 事件顺序：token* → sources → done

## 五、架构决策

1. **复用已有 TEI 服务**：Embedding 和 Reranker 已通过 TEI 部署（端口 8888/8889），后端通过 HTTP 调用，无需加载模型，无 GPU 依赖
2. **开发/部署双模式，先只做开发模式**：当前 Qdrant 用 `docker run` 单独启动（仅数据库本身），Backend 用 `uvicorn`，Frontend 用 `vite dev`，无 Docker Compose、无 Dockerfile；后续用户要求时再做容器化部署
3. **使用 LangChain / LlamaIndex 框架**：text_splitter、ChatOpenAI、QdrantVectorStore、LCEL 链式编排、ChatPromptTemplate 等全部复用框架能力，LangChain 和 LlamaIndex 按需选用
4. **Small-to-Big 两层检索**：小 chunk（≤500 字符）做 Embedding 检索保证精准度，大 chunk（完整 h2 section）喂 LLM 保证上下文完整；Qdrant 中分两个 collection 存储，检索后多一次主键查询（毫秒级）完成映射
5. **Small-to-Big + 按内容类型分治**：四种内容类型（术语定义/代码块/列表/叙述）各自有独立的切分策略，小 chunk 做检索、大 chunk 喂 LLM，兼顾检索精准度和生成上下文完整性
